Un seul décorateur de surface — @PublicExternalRoute() encapsule toute la logique d'auth, de guards et de scopes pour les routes externes. Les décorateurs bas niveau (@IsPublic(), @RequireAuth()) restent des détails d'implémentation internes.
@PublicExternalRoute({ authType, scope })
Seul décorateur à poser sur les controllers externes. Injecte localement :
ApiVersionSunsetGuard — endpoint retiré ?
ApiKeyIdentityGuard — résout la clé
ScopeGuard — scope vérifié
Pose @IsPublic() — toujours, car @PublicExternalRoute() est par définition une route externe accessible sans JWT. 
export function PublicExternalRoute(options: {
  authType: 'api-key' | 'hybrid',
  scope?: string,
}) {
  const guards = [ApiVersionSunsetGuard, ApiKeyIdentityGuard, ScopeGuard];
  return applyDecorators(
    IsPublic(), // toujours — @PublicExternalRoute est toujours une route externe
    SetMetadata(AUTH_TYPE_KEY, options.authType),
    SetMetadata(SCOPE_KEY, options.scope),
    UseGuards(...guards),
  );
}
authType
'api-key' — API Key obligatoire. Routes purement programmatiques. Cas le plus courant.

'hybrid' — JWT ou API Key. Routes de gestion accessibles depuis le dashboard ou par script (ex : créer sa première clé).

Coexistence avec le code existant — @Public() reste sur les controllers internes qui en ont besoin. Le test CI vérifie uniquement les controllers versionnés — les controllers internes ne sont pas dans le scope du test.
Sur l'API externe versionnée — pas sur l'API interne
Ces 3 endpoints sont sur /api/v1/... (version: '1'), pas sur l'API interne VERSION_NEUTRAL.

Point 2

Rate Limiting Quota

Algorithme — Sliding Window ✓
Standard moderne utilisé par la majorité des API Gateways. Calcule la consommation sur les N dernières secondes glissantes — élimine les pics de charge du Fixed Window.

Fixed Window — simple mais imparfait : 100 req à 14h59 + 100 req à 15h01 = 200 req en 2min sans être bloqué.
Token Bucket — plus fluide mais complexe à maintenir.
Sliding Window — compromis parfait, natif dans NestJS Throttler + Redis.
Tier-Based Bursting — deux paliers en parallèle ✓
Un développeur peut envoyer une rafale accidentelle (boucle mal synchronisée) puis plus rien pendant une heure. Avec un seul palier strict, il se prend des 429 dès la 2ème seconde.

La solution : deux fenêtres qui s'exécutent en parallèle. Une requête est bloquée si l'une des deux barrières est franchie :

Burst Rate — fenêtre courte (ex: 5 req/s) — empêche de paralyser le serveur instantanément
Sustained Rate — fenêtre longue (ex: 1000 req/min) — protège la capacité globale

Correspond aux paliers short + medium de NestJS Throttler — natif, rien à coder.
Segmentation ✓
Routes externes authentifiées → par clé API (bearer:{prefix}) ou par compte (account:{clientId})
Routes anonymes → par IP

Standard industrie : clé API pour les clients identifiés, IP pour l'anonyme (protection force brute sur login, inscription...).
Headers de réponse
Standard de fait (IETF Draft, adopté par GitHub, Twitter) — à renvoyer sur chaque réponse :

X-RateLimit-Limit: 1000 — quota de la période
X-RateLimit-Remaining: 985 — requêtes restantes
X-RateLimit-Reset: 1772124000 — timestamp Unix du reset

Permet au client d'adapter son débit proactivement sans attendre le 429.
Nestjs -> Override du throttlerGuard
Réponse 429 ✓
HTTP 429 Too Many Requests (RFC 6585) — jamais 403 ou 400.

Header obligatoire :
Retry-After: 60 — secondes à attendre avant de réessayer

Corps (RFC 7807 Problem Details) : 
{
  "type": "https://api.smappen.com/errors/rate-limit-exceeded",
  "title": "Too Many Requests",
  "status": 429,
  "detail": "Quota de 1000 req/heure dépassé. Reset dans 60 secondes."
}
Bucketing — proposition à valider
Routes externes → par clé API ou par client
Par clé : surcharger getTracker() → bearer:{prefix} si Authorization commence par sk_. Pas de résolution DB.
Par client : account:{clientId} — nécessite de résoudre la clé avant le bucketing (lookup DB ou cache Redis).

Routes anonymes → par IP
Comportement actuel du ThrottlerGuard global, inchangé.

À définir avec l'équipe produit.
Essential → short: ?, medium: ?   Pro → short: ?, medium: ?   Advanced → short: ?, medium: ?

Peut bloquer un client légitime qui envoie une petite rafale. Si tous les clients sont des scripts bien écrits, medium seul suffit peut-être.

Point 3

Crédits Facturation

Monnaie unique — le Crédit ✓
Standard de l'industrie (OpenAI, Shopify, Google Maps API). Au lieu de facturer à la requête, on abstrait via une unité commune — le crédit.

Chaque endpoint a un coefficient de coût basé sur sa consommation réelle CPU/DB :

@CreditCost(1) — route légère (lookup, liste)
@CreditCost(2) — géocodage
@CreditCost(10) — isochrone, calcul lourd

Absent → gratuit (routes de consultation, API interne)
Pre-flight check — HTTP 402 ✓
Le guard valide que le solde est suffisant avant d'exécuter la logique métier. Si insuffisant → HTTP 402 Payment Required instantané, le handler n'est jamais appelé.

402 est le code HTTP sémantiquement correct — "paiement requis". Jamais 403 ni 429.
Grille de facturation — standard industrie 2026 — à trancher
StatutFacturé ?Raison
2xxOuiValeur livrée
3xxNonSimple routage HTTP
401 / 403 / 429 / 402NonSécurité et quotas — évite les litiges
400 / 404NonTolérance DX — fautes de frappe, tests

4xx Rate Limiting — parade anti-abus
Si les erreurs sont gratuites, un attaquant peut bombarder l'API de requêtes invalides sans jamais vider son solde — pour saturer les serveurs ou scanner l'API.

Standard de fait : si un client enchaîne trop de 4xx sur une courte fenêtre, sa clé est temporairement bloquée (429).

Géré par ThrottlerGuard existant — pas de code supplémentaire si le bucketing par clé est adopté.
Headers de réponse ✓
Bonne pratique industrie — à renvoyer sur chaque réponse d'une route payante :

X-Credits-Cost: 2 — coût de la requête courante
X-Credits-Remaining: 8420 — solde après déduction

Permet au client de monitorer sa consommation sans appeler /usage à chaque fois.

Ces headers sont propriétaires — pas de standard IETF. Chaque provider nomme les siens différemment (OpenAI, Stripe...). La convention X-Credits-* est claire et cohérente.
Corps de réponse 402 ✓
Format cohérent avec les autres erreurs de l'API : 
{
  "errorKey": "api_credits_exhausted",
  "required": 10,
  "balance": 3,
  "detail": "Solde insuffisant. Achetez des crédits supplémentaires."

Approche DailyStat + Redis — zéro nouvelle table, réutilise l'infra Dania
Concept
Réutilise DailyStat (déjà en place pour Dania).

Pourquoi Redis ? DailyStat stocke un compteur qui monte — pour vérifier le quota, il faut lire, calculer, puis écrire (3 étapes séparées). Sous charge concurrente, deux requêtes peuvent lire le même solde et toutes les deux passer. Redis résout ça avec INCRBY — une opération atomique indivisible, comme un verrou sur la caisse.

C'est l'avantage de credit_account : UPDATE WHERE balance >= cost est nativement atomique en Postgres — pas besoin de Redis.
Stockage — deux couches
Redis — compteur temps réel du jour en cours. INCRBY atomique → pas de race condition.

DailyStat — historique longue durée, écrit en arrière-plan. domain = id de la clé appelante.

Total mensuel = SUM(DailyStat jours passés) + Redis aujourd'hui
DailyStat — domain = clé API, pas de constante fixe
Deux familles de tags, sans toucher au schéma :

product = 'api_credit_usage', domain = apiKey.id
→ pool de crédits — plafond sommé sur toutes les clés du client

product = 'api_requests:<catégorie>', domain = apiKey.id
→ comptage d'appels par catégorie — jamais dans le plafond

Grâce à la contrainte unique [client, domain, product, date] déjà en place, la répartition par clé et par catégorie est obtenue gratuitement — pas de colonne supplémentaire, pas de migration sur cette table.

// CreditGuard — compteur Redis atomique + flush async vers DailyStat
// Voir onglet ⚠️ Atomicité pour le détail du problème et de la solution
@Injectable()
export class CreditGuard implements CanActivate {
  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const { cost, domain } = this.reflector.getAllAndOverride<CreditCostMeta>(
      CREDIT_COST_KEY, [ctx.getHandler(), ctx.getClass()]
    ) ?? {};
    if (cost === undefined) return true; // no-op — route interne ou non taguée

    const request = ctx.switchToHttp().getRequest();
    const clientId = request.user.id;
    const apiKeyId = request.apiKey.id;
    const today = new Date().toISOString().slice(0, 10); // 'YYYY-MM-DD'
    const redisKey = `api:credits:${clientId}:${today}`;

    const limit = await this.clientsService.getApiCreditsLimit(clientId);

    // INCRBY atomique — Redis garantit qu'aucune race condition n'est possible
    const newValue = await this.redis.incrby(redisKey, cost);
    await this.redis.expire(redisKey, 60 * 60 * 48); // TTL 48h

    if (newValue > limit) {
      await this.redis.decrby(redisKey, cost); // rollback atomique
      throw new HttpException(
        { errorKey: 'api_credits_exhausted', required: cost, balance: limit - (newValue - cost) },
        HttpStatus.PAYMENT_REQUIRED,
      );
    }

    // Flush asynchrone vers DailyStat — historique long terme, non bloquant
    this.dailyStatsService
      .increment(API_CREDIT_USAGE_PRODUCT, clientId, apiKeyId, cost)
      .catch(err => this.logger.error('DailyStat flush failed', err));

    // Comptage par catégorie — analytics uniquement, n'affecte jamais le plafond
    if (domain) {
      this.dailyStatsService
        .increment(`${API_REQUEST_COUNT_PREFIX}${domain}`, clientId, apiKeyId, 1)
        .catch(err => this.logger.error('DailyStat category flush failed', err));
    }

    return true;
  }
}

Quand débiter les crédits ? bloquant
Option A — débit avant le handler
Simple. Si le handler échoue (500, timeout), le client est facturé pour un appel raté.

Option B — débit avant + remboursement sur échec
Plus juste, mais increment négatif sur DailyStat — risque de counter négatif si double remboursement. Fragile sans contrainte CHECK.

Option C — vérification avant, débit après sur succès uniquement
Pas de débit sur échec, pas de remboursement. Vérification et débit non atomiques — à gérer avec UPDATE WHERE used + cost <= limit au moment du débit. Compatible DailyStat.
Report des crédits non utilisés ?
Comportement par défaut : use-it-or-lose-it. Le report au mois suivant est une décision produit.
Quel historique dans /usage/history ? bloquant
L'implémentation de DailyStat dépend directement de ce choix — à trancher avant de coder.

Par client — total journalier toutes clés confondues. Simple, une ligne par jour. domain = constante fixe.

Par clé — répartition par clé API. Permet d'identifier une clé compromise ou sur-consommatrice. domain = apiKey.id.

Par feature — répartition par route (geocoding, isochrone…). Utile pour comprendre les usages. domain = nom de la feature.

Croisé clé × feature — "quelle clé a fait combien d'isochroness". Piste explorée : pas besoin de colonne supplémentaire ni de table dédiée — domain = apiKey.id pour le plafond, plus un second tag product = 'api_requests:<feature>' pour le comptage par catégorie, dans les deux cas avec la contrainte unique DailyStat existante. À valider — implique deux écritures par appel taggé plutôt qu'une, et un choix de convention de nommage pour les valeurs de product.

Les quatre options semblent implémentables sans modifier DailyStat ni migration — reste à confirmer que ça tient à l'usage avant de verrouiller le choix.

Point 4

Versioning Cycle de vie

URL Versioning — standard majoritaire
La version est dans l'URL. Simple, explicite, testable directement dans un navigateur.

GET /v1/users → GET /v2/users

Utilisé par GitHub, OpenAI, Twilio. NestJS le supporte nativement via enableVersioning().

Stripe est souvent cité comme exemple mais c'est un hybride : /v1/ dans l'URL (inchangé depuis 2011) + header Stripe-Version: 2026-06-24 pour l'évolution fine. Le système de transformation à la volée (Abstractors) derrière est extrêmement complexe — hors scope.
Header Versioning — puristes REST
L'URL représente une ressource, pas une version. La version passe dans un header.

Accept: application/vnd.smappen.v2+json
ou X-API-Version: 2

Utilisé par Microsoft, certaines APIs GitHub.

Problème DX — impossible de tester dans un navigateur, les logs ne montrent pas la version, le cache CDN nécessite Vary.
Règle 1 — ne versionner que les breaking changes ✓
Ajouter un champ optionnel → pas de nouveau version.
Supprimer un champ, changer son type, modifier la logique → nouvelle version.

Uniquement le majeur : v1, v2. Jamais v1.2.3 dans une URL d'API.
Règle 2 — politique de dépréciation
Deux headers complémentaires sur chaque réponse de la version dépréciée :

Deprecation: Thu, 19 Jun 2026 00:00:00 GMT — début de la période de dépréciation
Sunset: Wed, 11 Nov 2027 00:00:00 GMT — fin : date à laquelle la ressource devient indisponible
Link: <https://docs.smappen.com/migration/v2>; rel="successor-version"

Le client lit : "dépréciée depuis le 19 juin 2026, retirée le 11 nov 2027". Géré automatiquement par ApiVersionSunsetGuard — pose les headers sur les routes dépréciées et bloque avec 410 sur les routes retired: true. (RFC 8594 + IETF draft deprecation-header).
En pratique, les headers sont rarement lus...
Règle 3 — version globale de toute l'API ✓
Quand un breaking change a lieu, on bump toute l'API : /v1 → /v2. Les endpoints inchangés restent disponibles en /v2 via des alias — le client n'a qu'une seule variable de version à gérer.

GET /v1/* → toute la v1 dépréciée, headers Sunset posés auto sur chaque réponse
GET /v2/* → nouvelle version globale — certains endpoints ont changé, les autres ont le même comportement

Le client migre en changeant une seule variable : v1 → v2 dans sa base URL.

Standard de fait : Stripe, GitHub, OpenAI, Twilio. Une seule version à documenter à la fois.
@ApiVersion
Marque un endpoint comme actif — première version ou nouvelle version après une rupture.

version — numéro de version ('1', '2'...)
introducedAt — date de mise en service (documentation)

Se pose au niveau méthode. Hérite de la version du controller si absent. 
@Get()
@ApiVersion({ version: '1', introducedAt: '2026-06-19' })
listMaps() { ... }
@ApiDeprecated
Marque un endpoint comme déprécié — son successeur existe, une date de retrait est annoncée. Pose automatiquement les headers Deprecation et Sunset sur chaque réponse.

sunsetAt — date de retrait prévue
reason — explication du changement
migrationUrl — lien vers la doc de migration
retired: true — endpoint retiré, bloque avec 410 Gone 
// Déprécié — headers auto sur chaque réponse
@ApiDeprecated({
  version: '1',
  sunsetAt: '2027-01-01',
  reason: 'duration_s → durationSeconds',
  migrationUrl: 'https://docs.smappen.com/v2',
})
listMapsV1() { ... }

// Retiré — 410 Gone, une seule ligne change
@ApiDeprecated({ ..., retired: true })
Niveau d'application — les deux décorateurs se posent au niveau méthode. Le cycle de vie complet est @ApiVersion → @ApiDeprecated → @ApiDeprecated({ retired:true }) — seule la métadonnée change, le code métier reste intact tout au long.
Lien avec Swagger — ces décorateurs alimentent automatiquement la documentation OpenAPI :
@ApiVersion → endpoint visible dans le Swagger de sa version (/api/v1/docs)
@ApiDeprecated → endpoint barré dans Swagger UI (deprecated: true dans le JSON OpenAPI) + champ vendor x-api-version-info avec sunsetAt, reason, migrationUrl exploitables par un script CI
@ApiDeprecated({ retired:true }) → endpoint masqué du Swagger (plus de route active à documenter)
À trancher — le type de versioning n'est pas encore arrêté. Ce comparatif est là pour alimenter la décision.
URI versioning
GET /api/v1/geocode
GET /api/v2/isochrone
✓ Visible dans les logs sans config
✓ Testable curl / navigateur directement
✓ 404 explicite si version inexistante — contrat clair
✓ Chaque ressource évolue indépendamment
✓ Swagger : une seule config suffit
− Client doit connaître la version exacte disponible
− Change sa base URL à chaque migration
− URLs "moins propres" pour les puristes REST
Pas de fallback — /v2/geocode → 404 stable et prévisible, jamais la v1 en silence. Si une v2 apparaît plus tard, aucun client n'est surpris.
Header versioning
X-API-Version: 2
✓ URLs stables — la ressource ne change pas
✓ Client ne change pas sa base URL
✓ Sémantique HTTP stricte respectée
✓ Granularité possible par requête individuelle
− Invisible dans les logs sans enrichissement
− Non testable dans un navigateur
− Routing Traefik plus complexe
− Clients oublient souvent d'inclure le header
Utilisé par GitHub, Microsoft Azure, Salesforce.

Point 5

Scopes Permissions

Principe du Moindre Privilège Standard OAuth2 ∩ ABAC
Une clé d'API ne peut jamais accorder plus de droits que ce que le compte possède déjà. Les sessions JWT passent toujours — les scopes ne restreignent que les clés API.
🛡️ Le Scope (La Clé API)
Le Consentement (Forme) : Détermine les capacités techniques de la clé (ex: maps:read). Configuré et bridé par le client pour limiter l'impact en cas de fuite.
🗄️ CASL (L'Identité)
L'Habilitation (Fond) : Détermine l'ownership des données réelles de l'entreprise (ex: ID client). Géré par Smappen pour garantir l'étanchéité absolue de la DB.
Droits Finaux = Droits du Compte (CASL) ∩ Droits de la Clé (Scope)
La Syntaxe Standard : La notation ressource:action
maps:read -> Autorise uniquement la lecture ou le listing des cartes.
isochrones:write -> Autorise uniquement le calcul ou la création d'isochrones.
billing:read -> Autorise la lecture des factures, mais interdit de modifier les abonnements.
Corps de réponse 403 ✓
Format cohérent avec les autres erreurs de l'API : 
{
  "type": "https://api.smappen.com/errors/insufficient-scope",
  "title": "Forbidden",
  "status": 403,
  "detail": "Votre clé d'API ne possède pas les privilèges requis pour cette action.",
  "required_scope": "maps:write",
  "provided_scopes": ["maps:read", "usage:read"]
}
Scope via @PublicExternalRoute()
Le scope est passé directement dans @PublicExternalRoute() — @RequireScopes est un détail d'implémentation interne, jamais posé directement.

@PublicExternalRoute({ authType: 'api-key', scope: 'maps:read' })
@PublicExternalRoute({ authType: 'api-key', scope: 'analysis:run' })
Absent → no-op (API interne jamais affectée)
Impossible de l'oublier sur un controller externe
Puisque le scope est encapsulé dans @PublicExternalRoute(), et que le garde-fou CI vérifie que tout controller versionné porte ce décorateur, un scope manquant devient une erreur de build — pas un oubli silencieux en prod.
CASL + Scopes — deux guards séquentiels, pas une intersection
ScopeGuard
Vérifie que la clé a déclaré le scope requis à sa création.

maps:read présent → passe
maps:read absent → 403 immédiat

Ne connaît pas CASL. Ne connaît pas la ressource.
AuthorizationGuard (CASL)
Vérifie que le compte peut faire l'action sur la ressource.

map.clients.includes(userId) → passe
Sinon → 403

Ne connaît pas les scopes. Charge la vraie entité depuis la DB.
Résultat
Les deux s'exécutent en série via @UseGuards(). Si l'un échoue, l'autre n'est pas appelé.

Scope OK + CASL OK → accès
Scope KO → 403 (clé insuffisante)
Scope OK + CASL KO → 403 (ressource non possédée)

Pas d'intersection calculée — deux vérifications indépendantes.
1. ScopeGuard
Vérifie que la clé API a le droit d'appeler cette route.
Ex : la clé porte maps:read → elle peut appeler GET /v1/maps.
2. AuthorizationGuard (CASL)
Vérifie que le client est bien propriétaire de la ressource demandée.
Ex : map.client.id === userId → la carte appartient bien à ce client.
Les deux doivent être vrais
Avoir le scope ne suffit pas pour accéder à une ressource qui appartient à quelqu'un d'autre. Les deux vérifications s'appliquent en série.

Certains scopes n'ont pas de vérification CASL car la route ne touche pas à une ressource possédée (solde, calcul...).
Scope Ce que ça permet 1. ScopeGuard 2. CASL ownership
api-keys:manage Créer, lister, révoquer ses clés ✓ vérifié ✓ apiKey.client.id === userId
usage:read Consulter son solde et historique ✓ vérifié — pas de propriétaire
maps:read Lire ses cartes ✓ vérifié ✓ map.client.id === userId
datasets:read Lire ses datasets ✓ vérifié ✓ dataset.client.id === userId
datasets:write Modifier / supprimer ses datasets ✓ vérifié ✓ dataset.client.id === userId
data:read Données géo — extractor ✓ vérifié — pas de propriétaire (donnée partagée)
data:export Export CSV / XLSX ✓ vérifié — pas de propriétaire
analysis:run Calculs isochrone / analyse ✓ vérifié — pas de propriétaire (calcul)
Comment combiner ScopeGuard + AuthorizationGuard CASL
Pour une route externe qui touche une ressource possédée (ex : GET /api/v1/maps/:id), les deux s'appliquent en série — @RequireScopes au niveau classe (guard global), @UseGuards(AuthorizationGuard(...)) aussi au niveau classe (guard controller, exécuté après les guards globaux), @Can(Action.Read) au niveau méthode.

@Controller({ path: 'maps', version: '1' })
@Public()
@RequireAuth('apikey-only')
@RequireScopes('maps:read')                                         // 1. ScopeGuard — la clé a-t-elle le droit ?
@UseGuards(AuthorizationGuard(MapsService, MapV2, ['client'], 'mapId'))  // 2. CASL — la carte appartient-elle au client ?
export class MapsExternalController {
  @Get(':mapId')
  @Can(Action.Read)   // action lue par AuthorizationGuard
  @CreditCost(2)
  getMap(@Param('mapId') mapId: string) {
    return this.mapsService.findOne(mapId, ['client']);
  }
}
Bloquantes — à trancher avant de commencer
Les scopes sont-ils nécessaires ? CASL ne suffit-il pas ?
CASL seul — vérifie ce que le compte peut faire. Une clé compromise donne accès à tout ce que le compte permet. Simple — ScopeGuard disparaît.

CASL + Scopes — le client limite volontairement le périmètre d'une clé à la création. Une clé compromise n'expose qu'un sous-ensemble. Modèle Stripe "restricted keys".

Les deux sont orthogonaux — CASL vérifie l'ownership, les scopes vérifient le consentement explicite de la clé.
Clé sans scope — accès à tout ou à rien ?
Whitelist (recommandé) — une clé sans scope n'accède à rien. Plus sécurisé.

Accès large implicite — scope absent → ScopeGuard no-op → accès à tout.
Scopes wildcard — supportés ou non ?
Non supporté — chaque scope déclaré explicitement. Simple.

Wildcard — data:* couvre data:read + data:export. Nécessite une logique de matching dans ScopeGuard.
Importantes — à définir avant la mise en production
Scope supérieur à ce que le compte permet ?
Rejeté à la création ou seulement au moment de l'appel ?
Modification des scopes d'une clé existante ?
Ou faut-il révoquer et recréer ? Impact sur un éventuel PATCH /api/v1/api-keys/:id.

Lead Attribution — premier endpoint public

Besoin
Un client Smappen reçoit un nouveau lead, une commande ou une demande. Il a besoin de savoir automatiquement à quelle zone géographique (territoire franchisé, secteur commercial...) ce lead appartient, pour :

Endpoint
GET /api/v1/maps/:mapId/areas/locate

Prend une adresse en entrée, géocode en lat/lon, retourne les zones de la map qui contiennent ce point. 
GET /api/v1/maps/map_abc123/areas/locate
Authorization: Bearer sk_live_...
?address=12+rue+de+la+Paix+75001+Paris

→ 200 OK
X-Credits-Cost: 2
X-Credits-Remaining: 8418
Réponse
Retourne les zones qui contiennent l'adresse géocodée.
{
  "areas": [
    {
      "uuid": "zone_xyz789",
      "label": "Secteur Paris Centre"
    }
  ],
  "geocoded": {
    "address": "12 rue de la Paix, 75001 Paris",

JwtAccessAuthGuard — @IsPublic() + pas de JWT skip ✓
ThrottlerGuard — bucket bearer:sk_live_ ✓
ApiVersionSunsetGuard — v1 active ✓
ApiKeyIdentityGuard — résout sk_live_...
→ pose request.user = { id: clientId }
→ pose cls.set('apiKey', { id, scopes })
ScopeGuard — clé a areas:read ✓
AuthorizationGuard — charge MapV2[:mapId]
→ map.clients.includes(clientId) → ownership ✓
→ cls.set('client', { id, features, apiCreditsLimit })
CreditGuard — @CreditCost(2)
→ limit = cls.get('client.apiCreditsLimit') — 0 SELECT
→ INCRBY api:credits:clientId:date 2 → passe ✓
Handler locate()
→ géocode adresse → lat/lon
→ ST_Contains(geometry, ST_Point(lon, lat)) → retourne zones ✓
Requêtes DB
1. api_key — lookup par prefix + hash (ApiKeyIdentityGuard)
2. client + subscription — populate (AuthorizationGuard)
3. map_v2 + clients — ownership check (AuthorizationGuard)
4. area — filtre spatial PostGIS ST_Contains (Handler)

@Controller({ path: 'maps', version: '1' })
@PublicExternalRoute({ authType: 'api-key', scope: 'areas:read' })
@UseGuards(
  AuthorizationGuard(MapsService, MapV2, ['clients', 'metadatas'], 'mapId'),
  CreditGuard,
)
export default class PublicMapsController {

  @Get(':mapId/areas/locate')
  @Can(Action.Read)
  @CreditCost(2)
  @ApiVersion({ version: '1', introducedAt: '2026-06-25' })
  async locate(
    @Param('mapId') mapId: string,
    @Query() { address }: LocateAreaQueryDto,
  ): Promise<AreaLocationResponseDto> {
    const areas = await this.areaService.findContainingPoint(mapId, address);
    return {
      areas: areas.map((a) => ({ uuid: a.uuid, label: a.label })),
      geocoded: areas.geocoded,
    };
  }
}

// LocateAreaQueryDto
export class LocateAreaQueryDto {
  @IsString()
  @ApiProperty({ example: '12 rue de la Paix, 75001 Paris' })
  address: string;
